# Vibe-Coder — План доработки до MVP

## Цель

Довести текущую версию до стабильного MVP: всё основное работает, не виснет, покрыто тестами. После этого — отталкиваться от MVP и двигаться к расширенному функционалу.

---

## Фаза 1. Стабилизация и багфиксы

### 1.1. Зависание при парсинге JSON

Проблема частично решена, но может вернуться. Когда приходит некорректный JSON в режиме `processing`, приложение повисает.

**Задача:** провести ревью и стресс-тестирование парсера — убедиться, что любой невалидный JSON обрабатывается gracefully, без зависаний.

### 1.2. Присоединённые файлы не попадают в JSON

Файлы отображаются как «присоединённые» в интерфейсе, но фактически не включаются в формируемый JSON. Возможно, логика прикрепления где-то потерялась.

**Задача:** найти и починить место, где файлы должны сериализоваться в JSON. Проверить всю цепочку: выбор файла → хранение → включение в запрос.

### 1.3. Ошибки парсинга → вывод тоже должен быть JSON

Когда элемент неправильно распарсен, vibe-coder сообщает об ошибке и кладёт результат в clipboard. Но делает это в произвольном формате, из-за чего система «сходит с ума».

**Задача:** все ответы парсера (включая ошибки) оборачивать в JSON-формат для единообразия.

### 1.4. Сбой режимов при вставке JSON из Claude Code

При вставке JSON из чата Claude Code система его не распознаёт, хотя ожидает JSON. Сбиваются все режимы, приходится отправлять пустое сообщение, чтобы вернуться в режим ответа.

**Задача:** разобраться с логикой определения режима и приёма JSON. Сделать так, чтобы вставка JSON корректно обрабатывалась или, как минимум, не ломала текущее состояние.

### 1.5. Текстовое поле не привязано к чату

При переключении между чатами содержимое текстового поля ввода не меняется — во всех чатах отображается один и тот же текст.

**Задача:** привязать состояние текстового поля к конкретному чату. При переключении — восстанавливать сохранённый черновик.

---

## Фаза 2. Тестирование

Много функциональности, которая не покрыта тестами. Есть ощущение, что ранее работавшие фичи молча ломаются — это недопустимо.

**Минимальный набор тестов:**

- Запись и корректность логов
- Сериализация/десериализация сообщений
- Предсказуемый парсинг JSON (включая edge cases)
- Прикрепление файлов — файл реально попадает в JSON
- Устойчивость к невалидному JSON (не виснет, не ломает режим)
- Переключение чатов — черновик сохраняется

**Цель:** зафиксировать текущее поведение тестами, чтобы регрессии ловились автоматически.

---

## Фаза 3. Рефакторинг и архитектура

Проверить соответствие чистой архитектуре. Проблема с файлами, которые не попадают в JSON, может быть симптомом более глубоких нарушений в архитектуре.

**Задача:** провести ревью, при необходимости — рефакторинг слоёв, чтобы данные корректно проходили через всю цепочку.

---

## Фаза 4. Новый функционал (в рамках MVP)

### 4.1. Галочка «Включать историю сообщений»

При общении в одном и том же чате Claude не всегда нужно тянуть полную историю. Нужна возможность одной кнопкой/галочкой отключить отправку истории — чтобы продолжить с чистого листа, не пересоздавая чат.

### 4.2. Кнопка копирования JSON (уже сделана)

Кнопка для ручного копирования JSON из clipboard — уже реализована. Убедиться, что работает стабильно.

## Фаза 5. Пост-MVP (следующая итерация)

### 5.1. Планирование с автосозданием чатов

Режим планирования, который сам создаёт чаты под задачи.

### 5.2. Коммиты

Сейчас работает плохо: текст коммита генерируется к каждому JSON, что избыточно. Варианты решения: галочка «запрашивать коммит», ручной запуск коммита, или другая логика. Возможно, есть техническое ограничение на вставку данных в поле коммита — нужно исследовать.

### 5.3. Оптимизация токенов

Входящие токены, возможно, забираются с перебором. Выходящие уже минимизированы. Отслеживать по истории использования и оптимизировать по мере накопления данных.

### 5.4. Поддержка других языков и IDE

После стабилизации MVP — расширение на Python, Android Studio и другие среды.
